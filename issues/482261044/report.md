### 1. Maglev Graph Build

When processing the `AddSmi` bytecode, `GetInt32()` is called based on feedback to convert the `n41` node to the `Int32` type. This generates a `42: CheckedSmiUntag [n41]` to ensure that node `n41` is of type `SMI`, thus `NodeInfo(n41)->type=kSmi`.

Later, when processing `JumpLoop`, since `n41` is an Input Var for the `32: φᵀ r0 (n22, n41)` node, the `MergePointInterpreterFrameState::MergeLoopValue()` method will merge the type of `n41` into the `n32` node, so `Phi(n32)->type_` is set to `kSMI`.

```
void MergePointInterpreterFrameState::MergeLoopValue(
    MaglevGraphBuilder* builder, interpreter::Register owner,
    const KnownNodeAspects& unmerged_aspects, 
    ValueNode* merged,    // n41
    ValueNode* unmerged    // n32
) {
  ...
  NodeType type =    // Smi
      unmerged_aspects.GetTypeUnchecked(builder->broker(), unmerged);
  unmerged = EnsureTagged(builder, unmerged_aspects, unmerged,
                          predecessors_[predecessors_so_far_]);
  result->set_input(predecessor_count_ - 1, unmerged);

  result->merge_post_loop_type(type);    // Phi(n32)->post_loop_type_ = kSMI
  // We've just merged the backedge, which means that future uses of this Phi
  // will be after the loop, so we can now promote `post_loop_type` to the
  // regular `type`.
  result->promote_post_loop_type();    // Phi(n32)->type_ = kSMI

  ...
}

```

**However, it is important to note that `Phi(n32)->type_ = kSMI` does not mean that the node's runtime representation is `kSMI`. It only indicates that the node will be treated as a `kSMI` in subsequent uses.** In fact, during the PoC execution, when `if(i==100)` is true, `n5: HeapNumber` is passed into `n41`, and after `JumpLoop`, `n5` is then passed into `n32`.

### 2. Maglev Phi Untag

Next, we enter the `Maglev Phi Untag` phase to process the `41: φᵀ r0 (n32, n5)` node, which has two inputs:

- For `32: φᵀ r0 (n22, n41)`, the result from its `static type` is used directly, yielding `kSmi`.
- For `5: InitialValue(r1)`: The type of this input node is unknown, and normally, optimization for this Phi node would be abandoned. However, because `maglev_hoist_osr_value_phi_untagging` is enabled and it is an OSR case, the optimization proceeds by ensuring that `n5` is of type `Int32` at the beginning of the loop.

```
MaglevPhiRepresentationSelector::ProcessPhiResult

  ...

  for (int i = 0; i < node->input_count(); i++) {
    ValueNode* input = node->input(i).node();
    if (input->Is<SmiConstant>()) {
      ...
```

Below is the log for processing the `n41` node. It can be seen that Maglev Phi Untag incorrectly assumes that the `input_reprs` for the `n41` node are all `Int32`.

```
...
Considering for untagging: n41
  + use_reprs  : {Int32} (all uses)
  + input_reprs: {Int32}
Untagging kinds: {KnownSmi, SpeculativeOSRValue}
  + intersection reprs: {Int32}
  => Untagging to Int32
...

```

Therefore, during the Maglev Phi Untag optimization, the `Phi(n32)->type_ = kSMI` information from the `static type` analysis is used, and it is incorrectly assumed that the `n32` node is of type `SMI`. As a result, the `n32` phi node is `untagged` as a `SMI` type.

### 3. Commit Bisect

According to the comment in `MergeLoopValue()`, `result->promote_post_loop_type()` only indicates what type the node will be treated as in subsequent uses, which makes `phi->type_` inaccurate. However, the Maglev Phi Untag optimization requires `phi->type_` to be sound, because the generated conversion node assumes the node's type is `phi->type_` without performing a runtime check.

I believe the vulnerability was introduced in commit `ca38e2ef66278787411d9c45951b91d0c09336d5`, which enabled Maglev Phi Untag to infer types based on the node's static type.

## REPRODUCE

poc.js:

```
function f0() {
    let smi = 0;
    // x cannot be represented as a Smi, but it can be represented as an Int32,
    // so it is a HeapNumber object.
    var x = 0x4e000000; 
    for (let i = 0; i < 200; ++i) { // Magelv OSR Optimize
        if (i == 100) {
            smi = x;
        }
        smi + 2;
    }
    return smi;
}

// leak addr of HeapNumber in release compiled v8
print(f0());


```
