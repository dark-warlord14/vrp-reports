## Description

**VULNERABILITY DETAILS**  

The javascript generated by ImageFetchTabHelper::GetImageDataByJs is unsafely created using the following format string.

std::string js =  

base::StringPrintf("**gCrWeb.imageFetch.getImageData(%d, '%s')", call\_id*,  

url.spec().c\_str());  

web\_state*->ExecuteJavaScript(base::UTF8ToUTF16(js));

The `url.spec().c_str()` parameter is an image URL given from web content directly.  

If an image URL containing a script, e.g., `/'alert(1)+'.png`, is downloaded through "Save Image" in a child frame, ImageFetchTabHelper injects crafted JavaScript code into the parent frame. That triggers UXSS in the parent frame.

**VERSION**  

Chrome Version: 88.0.4324.69+beta  

Operating System: iPhone 8 Simulator with iOS 14.3

**REPRODUCTION CASE**  

The page for testing is below.  

<https://alice.csrf.jp/2021/chromium_ios/uxss1/parent.html>

In this page, parent frame is a victim hosted by alice.csrf.jp, that page is protected by CSP for prohibitting script execution.  

Its child frame is Evil hosted by evil.csrf.jp, that frame has an <img> tag to `/uxss'+alert('UXSS!:'+document.domain)+'.png`.  

When you tap the image and choose "Save image" or "Copy image", UXSS is triggered on the parent victim page.

### mi...@chromium.org (2021-01-13)

I am ok with a single bug for now, at least until we figure out how to proceed with fixing this. (I'll update the summary.)

This feature was designed in go/ios-chrome-copy-image, but I wasn't very involved with it. I'm adding some of the folks who were.

Eugene, Mark, Justin: As reviewers of the "Copy Image" feature, how should we handle this case. It sounds like we specifically wanted to try to use the WebView to download the image, but we are using the main frame to download an image which could be from a child frame.

I can think of a few mitigations:
1. Fix/sanitize the URL before handing it to the main frame. I am not sure if this is trivial or not.
2. Download the image in the same WebFrame which the element exists. This would require some feature changes as the WebFrame isn't available right now to ImageFetchTabHelper.
3. Remove this download method. There are still two other ways to get the image - draw it into a canvas or download outside of WKWebView.
4. Send the image data with the initial context menu element details instead of the URL. There is already a 100ms timeout since this method is only meant to retrieve the image from cache, not actually re-download so I think this wouldn't create any noticeable delay, but would increase the size of the context menu payload for images significantly and it gets called a lot (on every tap/scroll down).

### mi...@chromium.org (2021-01-14)

I uploaded a CL is escape the URL here: crrev.com/c/2628024. This won't prevent the leaking of the URL. Migrating image fetch JS to isolated worlds would mitigate the leaking of the URL, but running the JS within the same frame as the image (instead of just the same WebView) would be even better.

I suggest we proceed in this way:
- Land URL escaping CL crrev.com/crrev.com/c/2628024 and merge to M89
- Migrate image_fetch.js to isolated world, target M90
- Then optionally (but it will improve the success of fetching images via canvas drawing or cache hits) update code to call getImageData on the WebFrame which the image resides in. The WebFrame id is returned with the context menu message, but the WebFrame is not currently piped through the context menu action sheet to the ImageFetch code.
