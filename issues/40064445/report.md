## Description

**VULNERABILITY DETAILS**

This bug can be triggered by cros\_camera\_service's mojo interface which can be accessed from arcvm and chrome browser. So this bug may cause arcvm escape and chrome browser privilege escalation.

This bug is an integer overflow bug when calculate calloc size on 32 bit userspace chromebooks, will cause calloc a smaller memory and following controllable oob write.

Attacker can control the malloc size + overwrite size + overwrite content, so it is a very good bug from my opinion.

The vulnerable call path is :  

Camera3DeviceOpsDelegate::ProcessCaptureRequest -> CameraDeviceAdapter::ProcessCaptureRequest ->  

ScopedCameraMetadata DeserializeCameraMetadata -> allocate\_camera\_metadata -> calculate\_camera\_metadata\_size

In calculate\_camera\_metadata\_size, the two arguments are uint32 from mojo ipc data:  

size\_t calculate\_camera\_metadata\_size(size\_t entry\_count,  

size\_t data\_count) {  

size\_t memory\_needed = sizeof(camera\_metadata\_t); <------ on 32 bit userspace chromebooks, sizeof is 32 bit  

// Start entry list at aligned boundary  

memory\_needed = ALIGN\_TO(memory\_needed, ENTRY\_ALIGNMENT);  

memory\_needed += sizeof(camera\_metadata\_buffer\_entry\_t[entry\_count]); <----- integer overflow  

// Start buffer list at aligned boundary  

memory\_needed = ALIGN\_TO(memory\_needed, DATA\_ALIGNMENT);  

memory\_needed += data\_count; // uint8\_t data <----- integer overflow  

// Make sure camera metadata can be stacked in continuous memory  

memory\_needed = ALIGN\_TO(memory\_needed, METADATA\_PACKET\_ALIGNMENT);  

return memory\_needed;  

}

And in DeserializeCameraMetadata -> add\_camera\_metadata\_entry -> add\_camera\_metadata\_entry\_raw

tag, type, data, data\_count of add\_camera\_metadata\_entry\_raw are from mojo ipc data:

static int add\_camera\_metadata\_entry\_raw(camera\_metadata\_t \*dst,  

uint32\_t tag,  

uint8\_t type,  

const void \*data,  

size\_t data\_count) {

```
if (dst == NULL) return ERROR;  
if (dst->entry_count == dst->entry_capacity) return ERROR;  
if (data_count && data == NULL) return ERROR;  

size_t data_bytes =  
        calculate_camera_metadata_entry_data_size(type, data_count);  
if (data_bytes + dst->data_count > dst->data_capacity) return ERROR;  <------dst->data_capacity can be very big value, but dst buffer is small  

size_t data_payload_bytes =  
        data_count \* camera_metadata_type_size[type];  
camera_metadata_buffer_entry_t \*entry = get_entries(dst) + dst->entry_count;  
memset(entry, 0, sizeof(camera_metadata_buffer_entry_t));  
entry->tag = tag;  
entry->type = type;  
entry->count = data_count;  

if (data_bytes == 0) {  
    memcpy(entry->data.value, data,  
            data_payload_bytes);  
} else {  
    entry->data.offset = dst->data_count;  
    memcpy(get_data(dst) + entry->data.offset, data,  
            data_payload_bytes);       <---------- memcpy from controllable offset, data, payload_bytes  
    dst->data_count += data_bytes;  
}  
dst->entry_count++;  
dst->flags &= ~FLAG_SORTED;  
assert(validate_camera_metadata_structure(dst, NULL) == OK);  
return OK;  

```

}

**VERSION**

latest code of chromeos platform2/camera

Because my 32 bit userspace chromebook is not very good to build test images,

1, build eve or your 32bit chromebook test image by attached diff  

2, build chromium browser by attached diff, which will trigger mojo ipc to crash cros\_camera\_service  

3, after test image and chromium browser deploied, open the built-in camera app, the cros\_camera\_service will crash
