## Description

**VULNERABILITY DETAILS**  

Specifically crafted HTML file can trigger Use After Free of Device object in GPU process. This bug can potentially be exploited to achieve one click Remote Code Execution in a privileged process (i.e. the GPU process), or be exploited to escape sandbox from a compromised renderer process to GPU process.

When requestDevice() is called in JS, a backend C++ object Device (dawn::native::d3d12::Device) is created in GPU process to manage a whole bunch of Windows D3D stuff.  

Device maintains a linked list of ExternalImageDXGIImpl which are external image resources opened using this device:

```
class Device final : public DeviceBase {  
...  
	LinkedList<ExternalImageDXGIImpl> mExternalImageList;  
};  

```

ExternalImageDXGIImpl holds a direct raw pointer reference to Device:

```
class ExternalImageDXGIImpl : public LinkNode<ExternalImageDXGIImpl> {  
...  
  private:  
	Device\* mBackendDevice;  

```

The code inside ExternalImageDXGIImpl::Destroy() seems to be based on an assumption that whenever ExternalImageDXGIImpl is in the list, the Device object is alive and the mBackendDevice pointer is valid:

```
void ExternalImageDXGIImpl::Destroy() {  
	if (IsInList()) {  
		RemoveFromList();  
		...  
		mBackendDevice->ConsumedError(mBackendDevice->NextSerial());  
		mBackendDevice->ReferenceUntilUnused(mD3D12Fence.Get());  
		...  
	}  
}  

```

At first sight this assumption seems true, cause Device's destructor calls DeviceBase::Destroy(), which in turn calls Device::DestroyImpl(). In Device::DestroyImpl(), each ExternalImageDXGIImpl is removed from mExternalImageList:

```
void Device::DestroyImpl() {  

	while (!mExternalImageList.empty()) {  
		ExternalImageDXGIImpl\* externalImage = mExternalImageList.head()->value();  
		// ExternalImageDXGIImpl::Destroy() calls RemoveFromList().  
		externalImage->Destroy();  
	}  

```

However, if GPUDevice.destroy() is called before GPUDevice.experimentalImportTexture() in JS, mState is set to State::Destroyed:

```
void DeviceBase::Destroy() {  
...  
	DestroyImpl();  
...  
	mState = State::Destroyed;  
}  

```

The ExternalImageDXGIImpl object constructed afterwards is added to the mExternalImageList list. But later the destroy operation triggered by page refresh is skipped because of the'Device's state (mState):

```
void DeviceBase::Destroy() {  
	if (mState == State::Destroyed) {  
		return;  
		...  
	}  

```

So, the Device object thinks there is nothing in the list, while the ExternalImageDXGIImpl itself thinks it's in the list (IsInList() returns true).

Then the Device object is freed:

```
GpuChannel::DestroyCommandBuffer()  
	WebGPUCommandBufferStub::~WebGPUCommandBufferStub()  
		WebGPUDecoderImpl::Destroy()  
			DawnWireServer::~DawnWireServer()  
				WireServer::~WireServer()  
					Server::~Server()  
						ServerBase::DestroyAllObjects()  
							Device::~Device()  

```

Finaly when the ExternalImageDXGIImpl object is destructed, mBackendDevice (dangling pointer to the freed Device object) is used and related methods are called in ExternalImageDXGIImpl::Destroy(), thus the Use After Free:

```
void ExternalImageDXGIImpl::Destroy() {  
	if (IsInList()) {  
		RemoveFromList();  
		...  
		mBackendDevice->ConsumedError(mBackendDevice->NextSerial());  
		mBackendDevice->ReferenceUntilUnused(mD3D12Fence.Get());  
		...  
	}  
}  

```

d3d12: Disallow creating external image after device destruction

ExternalImageDXGIImpl holds a weak reference to the device. If the
device is destroyed before the image is created, the image will have a
dangling reference to the device which can cause a use-after-free.

This CL fixes that by adding a ValidateIsAlive() check before creating
the image similar to creating other API objects.
