## Description

**VULNERABILITY DETAILS**  

Certain methods of the Chrome OS chrome.wallpaper and chrome.wallpaperPrivate APIs can be used to access more than should be allowed.

--- chrome.wallpaper.setWallpaper() path traversal ---  

This method requires an extension or app with the "wallpaper" permission. As described in the documentation, it changes the user wallpaper using either the image at a supplied URL or supplied image file data (as an ArrayBuffer). The image is normally written into the following directory:  

/home/chronos/custom\_wallpapers/original/<user hash>

The filename that is used is supplied to the method as a string within the "details" object it accepts as a parameter. The problem is that this filename is never checked anywhere it is passed around:

1. In WallpaperSetWallpaperFunction::OnWallpaperDecoded(), the filename is retreived from the JS setWallpaper() call params as a string and passed to WallpaperControllerClient::SetCustomWallpaper() [1].
2. Passed to WallpaperController::SetCustomWallpaper() as file\_name [2].
3. Passed to WallpaperController::SaveAndSetWallpaper() as file\_name [3].
4. Passed to WallpaperController::GetCustomWallpaperPath() as file\_name (still a string), along with kOriginalWallpaperSubDir ("original") and wallpaper\_files\_id (user hash) [4]. Returned is a FilePath generated by appending (via Append()) wallpaper\_files\_id and file\_name to a FilePath representing /home/chronos/custom\_wallpapers (custom\_wallpaper\_path) [5]. If file\_name (directly controlled by JS) contains path traversal components (".."), the returned FilePath may represent a path above the normal directory. This FilePath is later used back within WallpaperController::SaveAndSetWallpaper() (as wallpaper\_path) to save the image to disk via SaveCustomWallpaper().

The consequence is that we're able to write valid image files to any directory that the chronos user has access to, using path traversal. The following example writes to /var/log/chrome/wp.jpg, with an image ArrayBuffer, arrayBuffer, and a wallpaper layout string ("STRETCH", "CENTER", or "CENTER\_CROPPED"), wallpaperLayout:  

chrome.wallpaper.setWallpaper({  

data: arrayBuffer,  

layout: wallpaperLayout,  

filename: '../../../../../var/log/chrome/wp.jpg'  

}, function () {});

("../../../../../" takes you to the root directory, as /home/chronos/custom\_wallpapers/original/<user hash>/../../../../../ translates to /.)

To verify that the file was written, you could either visit the corresponding file:// link within the browser (e.g. file:///var/log/chrome/wp.jpg) if you chose a directory that the browser can read from, or you could navigate to the directory with a developer mode shell if you chose one that it cannot.

The filename can include special characters and can be up to 255 characters long. If a file exists at the specified path, it will be overwritten. The above example writes to the log file directory, but we could also overwrite /home/chronos/user files like Preferences or History, or downloads, for example.

This is limited by the fact that the supplied image file is encoded as JPEG before it is written to the disk. However, the written data can still be predictable: the attached wp.jpg file (a 150x150 px blue square) is already encoded with the correct format, so its hash will be the same after it is written to the disk. Additionally, user files like Preferences, other extensions' data, and Linux container data should definitely not be (over)writable by wallpaper manager extensions, even if in a limited way.

--- chrome.wallpaperPrivate.setCustomWallpaper() path traversal ---  

The wallpaperPrivate API is the private version of the wallpaper API that only the official Wallpaper Picker app can use. It includes additional methods. This method works much like its public counterpart, but takes a few more parameters. It is also affected.

This should write to /var/log/chrome/wp.jpg, with an ArrayBuffer, arrayBuffer, and a wallpaper layout string, wallpaperLayout:  

chrome.wallpaperPrivate.setCustomWallpaper(  

arrayBuffer,  

wallpaperLayout,  

false, // generateThumbnail  

'../../../../../var/log/chrome/wp.jpg',  

false, // previewMode  

function () {}  

);

This is less critical than the previous issue, but Chrome OS profiles are still supposed to be restricted to a small subset of directories (e.g. ~/Downloads), as per <https://crbug.com/836362>.

Like wallpaper.setWallpaper(), behind the scenes, this method uses WallpaperControllerClient::SetCustomWallpaper(), as described above. It is called from WallpaperPrivateSetCustomWallpaperFunction::OnWallpaperDecoded() [6].

There is also a wallpaperPrivate.setWallpaper(), which similarly takes an ArrayBuffer of image data, but takes a URL string rather than a filename. The filename used is extracted from the last component of the URL (<https://example.com/images/image.jpg> -> image.jpg). As such, it appears that it is not affected.

--- chrome.wallpaperPrivate.getLocalImageData() arbitrary file read ---  

This issue isn't related to path traversal, and is the least critical. This method is intended to return the content of an image file on the disk at the given path as an ArrayBuffer. It doesn't just work with images, however. It also isn't restricted to any specific directories, and can be used to obtain the contents of any file that the chronos user has read access to by passing it the full path of the file, even though the Wallpaper Picker app doesn't need this capability.

This should print the contents of /home/chronos/user/Preferences to the console:  

chrome.wallpaperPrivate.getLocalImageData('/home/chronos/user/Preferences', function (arrayBuffer) {  

var blob = new Blob([arrayBuffer]);  

var reader = new FileReader();  

reader.onload = function () {  

console.log(reader.result);  

};  

reader.readAsText(blob);  

});

You can verify that this file is not normally exposed to the user by visiting the corresponding file:// link. ERR\_ACCESS\_DENIED should display.

**REPRODUCTION CASE**  

Attached are the files for an extension with "wallpaper" permission that demonstrates the first chrome.wallpaper.setWallpaper() issue. All four files should be placed into a folder, then loaded as an unpacked extension from chrome://extensions. Clicking on the extension's icon will display a simple interface with an option for writing the wp.jpg within the extension files to /var/log/chrome/wp.jpg, and an option that opens file:///var/log/chrome/wp.jpg within the browser to verify that it worked.

As for the wallpaperPrivate issues, the easiest way to demonstrate is by launching the developer tools for the official Wallpaper Picker app (via chrome://inspect) and pasting snippets into the console.

The following is an extended version of the previously provided wallpaperPrivate.setCustomWallpaper() snippet that retreives the wp.jpg from ~/Downloads as an ArrayBuffer to use for writing that can be pasted into the console:  

chrome.wallpaperPrivate.getLocalImageData('/home/chronos/user/Downloads/wp.jpg', function (arrayBuffer) {  

chrome.wallpaperPrivate.setCustomWallpaper(  

arrayBuffer,  

'STRETCH',  

false, // generateThumbnail  

'../../../../../var/log/chrome/wp.jpg',  

false, // previewMode  

function () {}  

);  

});

The previously provided snippet for wallpaperPrivate.getLocalImageData() that prints the user Preferences file to the console works without modification.
