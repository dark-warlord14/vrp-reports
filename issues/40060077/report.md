## Description

**VULNERABILITY DETAILS**  

V8's deserializer normally rejects malformed input but it can be coerced into producing typed arrays with a non-resizable ArrayBuffer and the is\_length\_tracking or is\_backed\_by\_rab flags set.

**VERSION**  

V8 Version: 10.1.124.8-node.13 but looks present in ToT  

Operating System: n/a (cross-platform)

**REPRODUCTION CASE**  

Easiest to show with node's binding to v8::ValueDeserializer:

// ArrayBuffer with elements [3,5,7,11] followed by Uint8Array  

const {deserialize} = require("v8")  

const a = deserialize(new Uint8Array([255,15,66,4,3,5,7,11,86,66,1,2,0]))  

const b = deserialize(new Uint8Array([255,15,66,4,3,5,7,11,86,66,1,2,1]))  

a.byteLength // ok  

b.byteLength // crash in Builtins\_TypedArrayPrototypeByteLength

Analysis: ValueDeserializer::ReadJSArrayBufferView() calls typed\_array->set\_bit\_field(flags) without validating flags.

Builtins\_TypedArrayPrototypeByteLength() branches when the typed array is variable length.

CodeStubAssembler::LoadVariableLengthJSTypedArrayByteLength() then tries to load the length from the wrong place (I think - not 100% sure about this one.)

ValueDeserializer::ReadJSArrayBuffer() always seems to create a non-resizable ArrayBuffer anyway so maybe the flags should not be part of the wire format?

### jb...@chromium.org (2022-06-27)

The second byte of those inputs is 15, the value serializer wire format version. I'd guess that the Node version you used doesn't support wire format version 15, so it was thrown out. You might be able to get away with trying the same input as version 14 (or maybe 13) if the input doesn't actually contain anything changed in the newer version. Or of course by using a newer version of Node.

I agree that it looks like the flags field should have been validated for correctness and consistency with other serialized data when this was added in https://chromium-review.googlesource.com/c/v8/v8/+/3297708, but this was missed.
